Equibit User Guide

This document describes how to use a Equibit Node and the RPC client.

An Equibit Node consists of a full Bitcoin Core implementation and a Equibit Core implementation.

Getting Help

bitcoin-cli help

Currently there is one RPC client program (bitcoin-cli) that is used to connect to both the Bitcoin Core and the Equibit Core. To get a list of command line options for bitcoin-cli, use

./bitcoin-cli -help

Equibit Node Command Line Options

To get a list of all command line options supported by the Bitcoin Core and Equibit Core, run the command:

./bitcoind -help

To also get a list of the Bitcoin Core debugging command line options, run the command:

./bitcoind -help -help-debug

The -eb_* versions of these commands will also work. That is,

./bitcoind -eb_help -eb_help-debug

See below for getting help for RPC commands.

Configuring the Node to support RPC processing

If a node is running, the RPC (Remote Procedure Call) client program (bitcoin-cli) can be used to communicate with it provided it is configured appropriately. HTTP client programs such as curl can also be used to communicate with an Equibit node.

In order for an Equibit node to handle RPC requests, the following steps must be done.

  1. Start the node with the -server option for Bitcoin Core RPC communication.
  2. Start the node with the -eb_server option for Equibit Core RPC communication.
  3. Define the rpcuser and rpcpassword options in the bitcoin.conf file

The bitcoin.conf file is located in the Bitcoin Core data directory. The default value is

~/.bitcoin. It can be set with the -datadir option. Therefore, the default location of bitconn.conf is ~/.bitcoin/bitcoin.confg

The equibit.conf file is located in the Equibit Core data directory. The default value is

~/.equibit It can be set with the -eb_datadir option.

The syntax of the rpcpassword definition (to be placed in the bitcoin.conf file) is:

rpcpassword=any-sequence-of-characters

The syntax of the eb_rpcpassword definition (to be placed in the equibit.conf file) is:

eb_rpcpassword=any-sequence-of-characters

Equibit Node RPC Help

Assuming the Node is running, a RPC client program can request a list of supported RPCs using the help RPC for Bitcoin RPCs and eb_help RPC for Equibit RPCs.

The Equibit RPC listening port is different from the Bitcoin RPC listening port. Since the RPC client will default to using the Bitcoin RPC listening port, the Equibit RPC requests also require the -rpcport option to be specified with the Equibit RPC port.

If another client program is used to connect to the Node, then the port specified will determine which RPCs should be used.

Therefore, the following commands will list the Bitcoin Core RPCs and the Equibit RPCs respectively:

./bitcoin-cli help ./bitcoin-cli -rpcport=18331 eb_help

Getting detailed help on individual RPCs can be done by specifying help (resp. eb_help) followed by the name of the command. For example, to get detailed help on the getinfo (resp. eb_getinfo) RPC, use the following commands:

./bitcoin-cli help getinfo ./bitcoin-cli -rpcport=18331 eb_help eb_getinfo

Networks

The Bitcoin Core and the Equibit Core can connect to three different networks.

  • The default network that the Bitcoin Core will connect to is the Bitcoin network.
  • The default network that the Equibit Core will connect to is the Equibit network.
  • If the -testnet option is specified, then the Bitcoin Core will connect to the Bitcoin test network
  • If the -eb_testnet option is specified, then the Equibit Core will connect to the Equibit test network
  • If the -regtest option is specified, then the Bitcoin Core will connect to the Bitcoin regression test network
  • If the -eb_testnet option is specified, then the Equibit Core will connect to the Equibit regression test network

If the -regtest (resp. -eb_regtest) option is specified, then the Bitcoin Core (resp. Equibit Core) is run in regression test mode. When run in this mode, a special chain in which blocks can be solved instantly is used. This is intended for regression testing tools and app development.

The network used determines where all files generated by a Core are placed. The path of files generated will have the following prefix:

NetworkPath Prefix
Bitcoin Core NetworkBitcoin-data-directory
Bitcoin Core Test NetworkBitcoin-data-directory/testnet3
Bitcoin Core Regression Test NetworkBitcoin-data-directory/regtest
Equibit Core NetworkEquibit-data-directory
Equibit Core Test NetworkEquibit-data-directory/testnet3
Equibit Core Regression Test NetworkEquibit-data-directory/regtest

The Bitcoin-data-directory defaults to ~/.bitcoin. It can be set with the -datadir option

The Equibit-data-directory defaults to ~/.equibit. It can be set with the -eb_datadir option

Configuring Node Connectitivity

The following options can be used to control to which Equibit Nodes a given node will connect. The options that control Equibit Node connectivity are:

-eb_addnode=ip-list
Add a node to connect to and attempt to keep the connection open
-eb_connect=ip-list
Connect only to the specified node(s)
-eb_dns
Allow DNS lookups for -addnode, -seednode and -connect (default: true)
-eb_dnsseed
Query for peer addresses via DNS lookup, if low on addresses (default: true unless -connect)
-eb_forcednsseed
Always query for peer addresses via DNS lookup (default: 0)
-eb_onlynet=network-list
Only connect to nodes in network net (ipv4, ipv6 or onion)
-eb_seednode=ip-list
Connect to a node to retrieve peer addresses, and disconnect

If peer node addresses are specified via the -eb_connect option, then by default, DNS seed processing is not done.

Peer nodes specified by -eb_addnode processing are added to the nodes discovered by all other means.

Peer nodes specified by the -eb_seednode option are contacted exclusively to discover other peers. Once the peer node information has been transmitted to the node, the connection is ended.

eb_onlynet should be set to one or more of ipv4, ipv6 and onion. Connections will be limited to the specified network(s).

RPC Commands

For each Bitcoin Core RPC, there is a corresponding Equibit Core RPC. The Equibit Core RPC name is the Bitcoin Core RPC name with a eb_ prefix. For example, Bitcoin Core RPC getmininginfo corresponds to Equibit Core RPC eb_getmininginfo. In addition, several new RPCs have been added to the Equibit Core interface in order to support Equibit Core functionality.

Bitcoin RPCs are documented at Bitcoin Developer Reference

This documentation can be used for the corresponding Equibit RPCs, with the knowledge that:

  • The Equibit Core uses separate TCP/IP networks
  • The Equibit Core wallet and blockchain data is stored to separate files

Equibit Specific RPCs

The RPCs listed in the Equibit category of the eb_help output have no corresponding Bitcoin RPC.

They can be sub-categorized as follows:

Equibit Issuer Commands

  • eb_getnewissuer - Creates a new issuer
  • eb_getissuers - Lists the issuers added to the wallet
  • eb_authorizeequibit - Authorizes Equibits by the specified issuer

Creating Messages

The RPCs to create the messages are:

  • eb_p2pmessage
  • eb_multicastmessage
  • eb_broadcastmessage
Uploading Messages

The following RPCs can be used to load message up to the client:

eb_getmessage can be used to load the message corresponding to the specified hash.

eb_getmessages can be used to load messages that satisfy the specified selection criteria. The select criteria parameters are:

from(date[:time]) Filters out messages that are newer than the from date
to(date[:time]) Filters out messages that are older than the to data
type(name[,...]) Loads messages that have a type equal to one of those specified by the type parameter
asset(name[,...]) Loads messages that have an asset equal to one of those specified by the asset parameters. This parameter has no effect on Peer-to-Peer messages.
sender(hash[,...]) Loads messages whose sender is equal to one of those specified by the sender parameter
receiver(hash[,...])Loads messages whose receiver is equal to one of those specified by the receiver parameter. This parameter has no effect on Multicast and Broadcast messages

If no filtering parameters are specified, then all messages are loaded.

Deleting Messages

The following RPCs can be used to delete messages from a wallet:

eb_deletemessage can be used to delete the message corresponding to the specified hash.

eb_deletemessages can be used to delete messagethat satisty the specified selection criteria.

As a safety measure, if no filtering parameters are specified, then no messages will be deleted.

Equibit Debugging Commands

The eb_dumpwallet (resp. dumpwallet) only dump a sub-set of the data stored in the Equibit (resp. Bitcoin) wallet. The Equibit RPC eb_dumpwalletdb dumps all of the wallet's contents.
